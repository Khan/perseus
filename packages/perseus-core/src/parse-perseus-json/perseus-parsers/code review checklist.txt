3. Once you have analyzed the changes in the PR, give a decision about whether to approve or request changes.

Note:
Keep the purpose of the PR in mind when reviewing. If the code changes don't reflect the intended purpose, then the PR should be rejected. If you reject a PR, you will give a clear two to three sentence explanation as to why it was rejected and what needs to change.

Code Review Guidelines:
The changes should not include:
- Any hard coding: constants and configuration values should be used instead
- Comments that describe what is being done: comments should explain WHY something is being done

Where applicable, the changes should:
- Specify hacks, workarounds, and temporary fixes
- Mention pending tasks and todos
- Avoid multiple if-else statements
- Use framework features wherever possible instead of writing custom code
- Prefer maps/filter over loops
- Follow established repo practices

Maintainability/Supportability
- The change should require the least amount of effort to support in the near future. It should be easy to identify and fix a bug.
- Changes should follow modern react practices or the modern practices set by the repo

Readability
- Code should be self-explanatory, you should get a sense of reading a story while going through code
-- If the code is unnecessarily complicated, flag this and offer a suggestion for simplifying the solution
- Appropriate names are used for variables, functions, and classes
- If it is difficult to understand the code, then it should be refactored or comments should be added to make it clear why something is being done

Testability
- Code should be easy to test
- Use interfaces while talking to different layers
- Avoid static functions and singletons as these are not easily testable by mocks
- Tests should not have conditional statements in them
- Bug fixes should include an accompanying test - either an update to a test or introduce a new test to catch this bug in the future
- Tests should have only 1 assertion
- Tests should follow the Arrange / Act / Assert pattern

Debuggability
- Provide support to log the flow of control, parameter data, and exception details to find the root cause. Send these details to sentry
- Functions should have a single purpose.
-- If a function is too large or has multiple purposes, it should be broken down into smaller functions

Configurability
- Keep the configurable values in place (like the database), so that no code changes are required if the data is changed frequently

Reusability
- Where appropriate
-- Code should be DRY (Do not Repeat Yourself)
-- Code should follow SOLID principles
--- Single Responsiblity: A class or function should only have one reason to change
--- Open/Closed: Objects or entities should be open for extension but closed for modification
--- Liskov Substitution: Every subclass or derived class should be sustitutable for their parent class
--- Interface Segregation: A client shouldn't be forced to implement an interface it doesn't use or be forced to depend on methods it doesn't use
--- Dependency Inversion: Entities must depend on abstractions, not concrete implementations.
-- Code should separate concerns
- Consider reusable services, functions, and components
- Consider generic functions

Reliability
- Exception handling and cleanup resources

Extensibility
- Add enhancements with minimal changes to the existing code. One component should be easily replaced by another

Questions to consider
- What could be the reprecussions of this change?
- What are the downstream and upstream dependencies of the files being changed?
-- Will this change affect any dependencies?
- Are there additional files that should be updated to make the change more comprehensive?
