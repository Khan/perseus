# Categorizer Widget — Accessibility Evaluation

**Standard:** WCAG 2.2 (W3C Recommendation, October 2023)
**Rubric:** [/docs/accessibility-rubric.md](/docs/accessibility-rubric.md)
**Component:** `packages/perseus/src/widgets/categorizer/categorizer.tsx`
**Evaluator:** Claude (static code analysis)
**Date:** 2026-04-29
**Method:** Static analysis of source, rendered snapshots, and shared styles

---

## Overall Score: Non-Conformant

Five Level A or Level AA criteria fail outright. The core interaction — selecting a category for each item — is entirely unreachable by keyboard on desktop. Selection state is invisible to assistive technology on both platforms.

---

## Findings

### FAIL — 2.1.1 Keyboard (Level A)

The interactive cells on desktop render as `<div role="button">` with only an `onClick` handler and no `tabIndex` attribute. Without `tabIndex="0"`, these elements are never placed in the tab order and cannot receive keyboard focus. There are no `onKeyDown`/`onKeyUp` handlers either.

The codebase acknowledges this explicitly:
```tsx
// categorizer.tsx:164
{/* eslint-disable-next-line jsx-a11y/click-events-have-key-events,
    jsx-a11y/interactive-supports-focus -- TODO(LEMS-2871) */}
<div role="button" aria-label={catName} onClick={...}>
```

On mobile, the `<input type="radio">` inside each cell does support keyboard interaction, but the wrapping `<div role="button">` creates conflicting semantics.

**Fix:** Add `tabIndex={0}` and an `onKeyDown` handler that invokes `_handleUserInput` on Enter or Space. Better: replace the `<div role="button">` pattern with a proper `role="radio"` / `role="radiogroup"` pattern per row.

---

### FAIL — 2.4.3 Focus Order (Level A)

Consequence of the missing `tabIndex`: keyboard focus skips every categorization cell on desktop. The tab order jumps from outside the widget entirely to the next focusable element after it. There is no way to navigate between cells with Tab.

---

### FAIL — 2.4.7 Focus Visible (Level AA)

Because the desktop buttons are not focusable, they never display a focus indicator. Even if `tabIndex` were added today, there is no custom focus style defined — only a `:hover` color change — so the browser default outline (which Aphrodite may suppress via `outline: none` on the shared `responsiveInput` style) would need to be verified.

---

### FAIL — 4.1.2 Name, Role, Value (Level A)

Two problems:

**1. No selected-state communication.** Each cell button (`role="button"`) has no `aria-pressed`, `aria-checked`, or any other state attribute. When a selection is made, the icon changes from a thin circle to a filled circle and the color changes, but assistive technology receives no update. A screen reader user has no way to know which category is currently selected for any item.

The correct ARIA pattern for this widget is a per-row radio group:
- `<tr>` → `role="radiogroup"` labeled by the item cell content
- Each cell button → `role="radio"` with `aria-checked={selected}`

**2. Mobile radio inputs are unlabeled.** On mobile, real `<input type="radio">` elements are rendered inside the `<div role="button">`, but the inputs themselves have no `aria-label`, no `aria-labelledby`, and no associated `<label>` element. The label lives only on the parent `<div>`. Screen readers will announce the input as an unnamed radio button.

---

### FAIL — 4.1.3 Status Messages (Level AA)

When the user selects a category, there is no `aria-live` region or `role="alert"` that announces the change. A screen reader user clicks or activates a button and receives no feedback that their selection was registered.

---

### FAIL — 1.4.11 Non-text Contrast (Level AA)

The unselected icon color is `color: #ccc` (`#CCCCCC`) rendered on a white background. The contrast ratio is approximately **1.57:1**, well below the 3:1 minimum required for interactive UI components.

| State | Color | Background | Contrast | Result |
|-------|-------|-----------|----------|--------|
| Unselected | `#CCCCCC` | `#FFFFFF` | ~1.57:1 | **FAIL** |
| Selected | `#333333` | `#FFFFFF` | ~10.9:1 | Pass |
| Static selected | `#888888` | `#FFFFFF` | ~3.14:1 | Pass (barely) |

The unselected state is the resting state that users must perceive to understand what is selectable. A user with low vision may not be able to see the unselected icons at all.

**Fix:** Increase the unselected icon color to at minimum `#767676` (~4.5:1) or `#949494` (~3.0:1). A border on the cell would also help communicate the interactive target boundary.

---

### PARTIAL PASS — 1.3.1 Info and Relationships (Level A)

The `<table>` structure is partially correct but has several gaps:

- **No `<caption>`.** The table has no accessible name. A screen reader announces it as a generic table.
- **No `scope` attributes.** Column `<th>` elements lack `scope="col"`. The row-item cells are `<td>`, not `<th scope="row">`, so the row-to-item association is not programmatic.
- **Wrong ARIA pattern.** The grid represents a mutually exclusive selection per row — semantically a set of radio groups, one per item. Using `role="button"` loses the radio group semantics (mutual exclusivity, checked state, group label) that screen reader users expect.
- **Empty first header cell.** The top-left cell is `<td class="emptyHeaderCell">` with no content or label. This is acceptable for a visual spacer, but a `<th scope="col">` with `aria-label="Item"` or similar would be clearer.

---

### PARTIAL PASS — 1.4.3 Contrast (Minimum) (Level AA)

Text in category column headers (`<th>`) and item row cells renders at default browser text color on white — assumed to pass. However, the `staticCheckedRadioSpan` color `#888888` is used as a visual label for a selected answer in review mode. If any small text accompanies that icon at that color it would fail the 4.5:1 text threshold, but the icons themselves only need 3:1, which `#888888` meets at 3.14:1.

No specific text contrast failure confirmed in static analysis, but contrast of hint text, placeholder text, or any text rendered inside a `<Renderer>` at a reduced opacity should be verified in a live browser.

---

### PARTIAL PASS — 2.5.3 Label in Name (Level A)

Each `<div role="button">` has `aria-label={catName}`, e.g. `aria-label="No relationship"`. That accessible name correctly matches the visible column header text — pass for that element.

However, on mobile the `<input type="radio">` inside each button has no accessible name at all. Its visible label is only on the parent `<div>`. Speech input users saying "No relationship" would activate the `<div>`, but the radio input itself is unlabeled.

---

### PARTIAL PASS — 3.3.2 Labels or Instructions (Level A)

There are no instructions telling the user how to interact with the widget (e.g., "Select one category for each item"). Sighted users can infer this from the visual grid layout, but screen reader users navigating the buttons in linear order have no context for the overall task. A table `<caption>` or a visually-hidden instruction element before the table would address this.

---

### PARTIAL PASS — 5.5 Scoring and Feedback Accessibility (Perseus-specific)

Selection state is not communicated to screen readers (see 4.1.2 and 4.1.3 above). When correctness feedback is rendered by the parent, it's handled outside this widget. Within the widget, the filled-circle vs. thin-circle icon does convey selected vs. unselected through icon shape in addition to color — so 1.4.1 (Use of Color) passes — but the state change is never programmatically announced.

## Recommended Fix Priority

The fixes below prioritize native HTML semantics and Wonder Blocks tokens over hand-authored ARIA. The core principle: an `<input type="radio">` needs zero explicit ARIA — the browser exposes role, name, checked state, and keyboard behavior automatically. This is simpler and more robust than patching the existing `<div role="button">` approach.

Wonder Blocks packages available in this repo: `wonder-blocks-core` (`View`, `addStyle`) and `wonder-blocks-tokens` (`semanticColor`, `border`, `sizing`, `font`).

---

### P0 — Unify both platforms on native `<input type="radio">`

**Fixes:** 2.1.1, 2.4.3, 2.4.7, 4.1.2, 4.1.3, 2.5.3

The mobile path already renders `<input type="radio">`. Extend it to desktop and remove the `isMobile` branch entirely. A native radio input provides keyboard support (Tab to enter group, arrow keys to move within group), checked-state announcement, and mutual exclusivity — all without any ARIA.

Remove the `<div role="button">` wrapper and the `InlineIcon` span entirely. Create the input via `addStyle` from `wonder-blocks-core` and do **not** apply `sharedStyles.responsiveInput` — that style sets `outline: none` which suppresses the native focus ring. Define all styles fresh in the widget:

```tsx
import {addStyle} from "@khanacademy/wonder-blocks-core";
import {semanticColor, border} from "@khanacademy/wonder-blocks-tokens";

const StyledRadio = addStyle("input");
```

Associate each input with a real `<label>` that includes **both the item name and category name** so screen reader users who navigate by Tab (not by table) hear full context without needing to scan the table headers:

```tsx
const inputId = `${uniqueId}-cat-${catNum}`;
// item is the raw content string (e.g. "Graph 1"); catName is the category string.
// Using both gives the full context: "Graph 1 — No relationship, radio button, not checked"
<StyledRadio
    type="radio"
    id={inputId}
    name={uniqueId}
    checked={selected}
    disabled={this.props.static}
    onChange={() => this._handleUserInput(itemNum, catNum)}
    style={[
        styles.radioInput,
        selected && styles.radioInputChecked,
        this.props.static && styles.radioInputDisabled,
    ]}
/>
<label htmlFor={inputId} className={css(styles.srOnly)}>
    {`${item} — ${catName}`}
</label>
```

The `disabled` prop on native inputs is natively exposed to AT as a state — no `aria-disabled` needed. When `this.props.static` is true the browser announces "dimmed" or "unavailable" depending on the screen reader, which correctly communicates that the widget is in review mode.

**Why this closes 4.1.3 (Status Messages):** When a native radio button's `checked` state changes, the browser automatically announces the new state to AT without a separate `aria-live` region. Selection is announced as part of the value change on the focused element itself, not as a background status message — this is the correct and simpler path for a within-widget selection change.

**Focus visible (closes 2.4.7):** The widget's `styles.radioInput` must NOT include `outline: none`. The browser's native `:focus-visible` ring is sufficient. If the visual design requires a custom ring, add it explicitly:

```tsx
const styles = StyleSheet.create({
    radioInput: {
        // explicit focus ring — do NOT set outline: none here
        ":focus-visible": {
            outline: `${border.width.medium} solid ${semanticColor.core.border.focus.default}`,
            outlineOffset: border.width.thin,
        },
        // sizing: use circleSize directly (not circleSize - 2) to meet 24px minimum
        blockSize: circleSize,       // 24px
        inlineSize: circleSize,      // 24px
        // appearance
        appearance: "none",
        WebkitAppearance: "none",
        borderRadius: "50%",
        boxShadow: `0 0 0 ${border.width.thin} ${semanticColor.core.border.neutral.default}`,
        cursor: "pointer",
    },
    radioInputChecked: {
        boxShadow: `0 0 0 ${border.width.medium} ${semanticColor.core.foreground.instructive.default}`,
        backgroundColor: semanticColor.core.foreground.instructive.default,
    },
    radioInputDisabled: {
        cursor: "not-allowed",
        // semanticColor.core.foreground.neutral.weak is designed for disabled UI elements
        // and meets 3:1 non-text contrast requirement
        boxShadow: `0 0 0 ${border.width.thin} ${semanticColor.core.foreground.neutral.weak}`,
    },
    srOnly: {
        // standard visually hidden pattern
        position: "absolute",
        inlineSize: 1,
        blockSize: 1,
        overflow: "hidden",
        clip: "rect(0,0,0,0)",
        whiteSpace: "nowrap",
    },
});
```

**Target size (closes 2.5.8):** Using `blockSize: circleSize` (24px) instead of `circleSize - 2` (22px) ensures the input meets the 24×24px minimum. Do not inherit `responsiveInput` shared styles which subtract 2px.

### P1 — Fix unselected-state contrast using `semanticColor` tokens

**Fixes:** 1.4.11, 1.4.3

Replace all hardcoded color values with `semanticColor` tokens. The P0 styles above already handle the radio input colors. Additionally:

- Remove `styles.cell` color `#ccc` — this was the fill color for the icon; it is no longer needed after the icon is removed.
- Remove `styles.checkedRadioSpan`, `styles.staticCheckedRadioSpan`, `styles.radioSpan` — all removed with the icon.
- Remove `styles.emptyHeaderCell` border `2px solid #ccc` — replace with `border.width.thin solid ${semanticColor.core.border.neutral.subtle}`.

Token contrast guarantees:
- `semanticColor.core.border.neutral.default` — designed to meet ≥3:1 against the default background (non-text contrast, passes 1.4.11). **Verify the resolved hex value during implementation** by inspecting the token in a browser or checking the `wonder-blocks-tokens` package source to confirm it is ≥ `#949494` on white.
- `semanticColor.core.foreground.instructive.default` — the primary interactive selection color (passes both 1.4.11 at 3:1 and 1.4.3 at 4.5:1 if used for text)
- `semanticColor.core.foreground.neutral.weak` — the WB disabled foreground token; designed to meet ≥3:1 for non-text UI indicators

### P2b — Add `<caption>`, `scope` attributes, and row headers to the table

**Fixes:** 1.3.1, 3.3.2

Use `addStyle("caption")` from `wonder-blocks-core` so Aphrodite styles apply to the caption. Visually hide it to avoid redundancy with surrounding question text.

```tsx
const StyledCaption = addStyle("caption");

<table className={css(styles.categorizerTable)}>
    <StyledCaption style={styles.srOnly}>
        {this.context.strings.categorizerTableCaption}
    </StyledCaption>
    <thead>
        <tr>
            {/* corner cell: scope="col" with empty accessible name is valid for a spacer column */}
            <th scope="col" className={css(styles.emptyHeaderCell)} />
            {this.props.categories.map((category, i) => (
                <th scope="col" className={css(styles.header)} key={i}>
                    <Renderer content={category} ... />
                </th>
            ))}
        </tr>
    </thead>
    <tbody>
        {indexedItems.map(([item, itemNum]) => (
            <tr key={itemNum}>
                {/* Change from <td> to <th scope="row"> so AT associates
                    the item name with every cell in this row */}
                <th scope="row" className={css(styles.itemCell)}>
                    <Renderer content={item} ... />
                </th>
                {this.props.categories.map((catName, catNum) => {
                    const selected = this.props.userInput.values[itemNum] === catNum;
                    // uniqueId is already scoped per-item: `${self.state.uniqueId}_${itemNum}`
                    // Adding -cat-${catNum} makes each input's id unique across the entire table.
                    const inputId = `${uniqueId}-cat-${catNum}`;
                    return (
                        <td key={catNum} className={css(styles.cell)}>
                            <StyledRadio
                                type="radio"
                                id={inputId}
                                name={uniqueId}
                                checked={selected}
                                disabled={this.props.static}
                                onChange={() => this._handleUserInput(itemNum, catNum)}
                                style={[
                                    styles.radioInput,
                                    selected && styles.radioInputChecked,
                                    this.props.static && styles.radioInputDisabled,
                                ]}
                            />
                            <label htmlFor={inputId} className={css(styles.srOnly)}>
                                {`${item} — ${catName}`}
                            </label>
                        </td>
                    );
                })}
            </tr>
        ))}
    </tbody>
</table>
```

The corner `<th scope="col" />` is valid HTML for a table spacer cell. Screen readers skip empty header cells.
